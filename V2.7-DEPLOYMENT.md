# X FITNESS v2.7.0 — Security Hardening Release

## What changed and why

Before v2.7, the customer-facing `/checkin/*` pages used the `anon` Supabase
key, which is visible in the browser to anyone who opens DevTools. The
database had a wide-open RLS policy that let anyone with that key download
every customer's IC, name, phone, emergency contact, and guardian information
via a single `curl` call. This is a PDPA-grade data leak waiting to happen.

v2.7 closes this. Anonymous traffic from `/checkin/*` can now only:

- INSERT new customers (during registration), with privileged fields
  automatically stripped server-side.
- INSERT visits (during check-in), with timestamps forced to `now()`.
- Read non-PII columns of the `visits` table (for the 30-minute cooldown
  message), filtered by IC the caller already knows.
- Look up a customer by **exact** IC via `lookup_customer_for_checkin()` RPC
  — this returns only the minimal fields needed for the check-in flow
  (id, nationality, ic, name, dob, status, membership, gender). Phone,
  emergency contact, and guardian data are no longer exposed to anon.
- Look up a customer by **exact** phone via `lookup_customer_by_phone()` RPC
  — returns only id and status, used by `/register` to detect duplicate phone.

What anon **can no longer do**:

- ❌ `select * from customers` (returns "permission denied")
- ❌ Enumerate customers by name, partial IC, or phone pattern
- ❌ Self-promote a registration to membership=member
- ❌ Inject arbitrary text into customer notes
- ❌ Set status=banned or warning_count or ban_reason on themselves
- ❌ Backdate visit records

Authenticated staff/admin paths are completely unchanged — they have
full access exactly as before.

There is also a daily rate limit (500 customers, 3000 visits per 24h)
to cap damage from any flooding attack. These ceilings are well above
realistic gym volume and won't impact normal operation, including
admin Excel imports.

## Deployment steps

The migration is **idempotent and safe to run in production**, but
because it changes RLS policies and column grants, you should deploy
the SQL and frontend together.

### Step 1 — Run the SQL migration

Open Supabase SQL Editor (https://app.supabase.com/project/_/sql) and
run **`migration-v2.7-security-hardening.sql`** in full.

You should see a NOTICE block at the end like:

```
NOTICE: ✓ v2.7 security hardening installed
NOTICE:   - Removed public SELECT on customers
NOTICE:   - Created lookup_customer_for_checkin() RPC
NOTICE:   - Created lookup_customer_by_phone() RPC
NOTICE:   - INSERT sanitization triggers active
NOTICE:   - Daily rate limits: 500 customers / 3000 visits per 24h
```

### Step 2 — Deploy the frontend

Push the updated code to your GitHub repo as usual. Vercel will
auto-deploy. Files changed in this release:

- `app/checkin/id-input/page.tsx` — uses RPC for customer lookup
- `app/checkin/register/page.tsx` — uses RPCs for phone/customer lookup
- `app/checkin/reminders/page.tsx` — visits query filters by `ic` (not `customer_id`)
- `migration-v2.7-security-hardening.sql` — the new migration
- `package.json` — version bump to 2.7.0

### Step 3 — Verify

After deployment:

1. Open `/checkin` on your phone, walk through register and check-in.
   The flow should look and feel exactly the same.
2. As a sanity check, open DevTools → Network tab on the customer screen.
   You'll see calls to `/rest/v1/rpc/lookup_customer_for_checkin` instead
   of `/rest/v1/customers?select=*`.
3. (Optional) From a terminal, try the old attack:
   ```bash
   curl 'https://YOUR-PROJECT.supabase.co/rest/v1/customers?select=*' \
     -H "apikey: YOUR_ANON_KEY"
   ```
   It should return `{"code":"42501","message":"permission denied for table customers"}`.

### Rollback

If something breaks, the migration can be undone by running:

```sql
-- Restore the old (insecure) public SELECT policy
create policy "Public can read for check-in"
on customers for select to anon
using (true);

-- Re-grant default anon privileges
grant all on customers to anon;
grant all on visits to anon;

-- Drop the new RPCs and triggers
drop function if exists lookup_customer_for_checkin(text);
drop function if exists lookup_customer_by_phone(text);
drop trigger if exists trg_sanitize_anon_customer_insert on customers;
drop trigger if exists trg_sanitize_anon_visit_insert on visits;
drop trigger if exists trg_enforce_anon_customer_daily_limit on customers;
drop trigger if exists trg_enforce_anon_visit_daily_limit on visits;
drop function if exists sanitize_anon_customer_insert();
drop function if exists sanitize_anon_visit_insert();
drop function if exists enforce_anon_customer_daily_limit();
drop function if exists enforce_anon_visit_daily_limit();
```

But the old frontend would also need to be re-deployed since the new
frontend uses RPCs that don't exist after rollback.

## Limitations

This release does NOT add:

- **CAPTCHA / Turnstile** — Cloudflare Turnstile is free and would close
  the remaining "high-volume automated registration" gap (someone could
  still register up to 500 fake customers per day from real-looking
  browsers). If gym signups exceed 50/day, consider adding this.
- **IP-based blocking** — Supabase free tier doesn't expose client IP
  in trigger context without extra setup. The 24h rate limit caps
  damage but doesn't identify the attacker.
- **Audit logging of anon activity** — only authenticated actions are
  logged in `audit_log`. Adding anon-side logging would help post-incident
  forensics but increases write volume.

These are reasonable next steps if traffic grows.
